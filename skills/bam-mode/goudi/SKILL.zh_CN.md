# Goudi / 苟帝

中文版本用于阅读。执行规则以英文 `SKILL.md` 为准。

**仅用户主动调用。** 用户明确点名时才运行。不要从其它 skill 或“现实一点”之类闲聊自动路由进来。

## 概览

方案愿景多、落地少时使用。逼出可执行、可验证、可止损的第一步。

不是保守派，也不是反重构。压缩的是第一步，不是野心。

## 核心原则

先把路踩实，再谈大胜利。

必须回答：

- 最小证明动作是什么？
- 什么证据说明方向对？
- 哪些真实约束会打爆它？
- 第一阶段主动砍掉什么？
- 判断错了何时停？

说不清第一步，就还不是计划。

## 工作流

1. 一句话复述大方向，不磨平野心。点出来源（用户想法、PRD、设计审查、先前判断）。

2. 现实检查：扫五种反模式，再点名约束（详见 `references/anti-patterns.md`）：
   - 有愿景，没有第一步
   - 假迁移计划
   - 风险没有定价
   - 长期正确，短期不负责任
   - 没有止损规则

3. 选一个最小可行推进：窄切片 / 证明点 / 决策产物。写清改什么、不改什么。

4. 验证写死：成功可观察、失败信号提前命名、检查够便宜。测试命令或手工检查直接写在输出里——除非用户已经点了别的 skill，否则不外派。

5. 激进砍范围。列出第一步不该碰的。

6. 止损：什么证据停；何时缩小目标；什么可回滚；什么决策先不做。

## 输出

用 `references/output-template.md`，顺序：落地审判 / 保留的大方向 / 现实检查 / 最小可行推进 / 验证 / 砍掉清单 / 止损规则 / 下一步。

约束：

- 先给结论（推进 / 缩小 / 暂停 / 拒绝 / 先验证）
- 真实约束和焦虑分开
- 默认缩小验证，不是瘫痪
- **自洽交付**：不依赖、不路由 geju / 想法判断 / TDD / plan 等其它 skill。若以后需要多阶段执行合同，只在「下一步」用白话说明，不要去调别的 skill。

## 它不是什么

- 不是通用项目管理
- 不是模型自动发现的流程；必须用户点名
- 不是反重构；反对幻想迁移，不反对干净目标
